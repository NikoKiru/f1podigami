# Mobile Experience Redesign — Design Spec

**Date:** 2026-06-16
**Status:** Approved (design), pending implementation

## Goal

Make all four pages of the F1 Podigami static site beautiful and genuinely usable on
mobile phones after deployment. Eliminate sideways scrolling, give each page a layout
tuned to small screens, and keep tap targets and input sizing thumb-friendly.

## Constraints & Approach

- **CSS-only.** The HTML is generated by Python build scripts but the CSS files
  (`style.css`, `index.css`, `seasons.css`, `soulmates.css`, `charts.css`) are static
  and linked directly. Every table cell already carries a stable class hook
  (`td.rank`, `td.drivers`, `td.count`, `td.last`; `.col-r`, `.col-name`,
  `.col-drivers` on seasons). Card labels come from CSS `::before` content, so **no
  build-script changes and no HTML regeneration are required.** Changes survive future
  rebuilds.
- **Breakpoints:** primary table→card flip at `max-width: 600px`. Existing page
  breakpoints (720/760/960/1000px) are retained/refined for intermediate sizing.
- Viewport and `theme-color` meta tags already exist on every page — no change needed.

## Cross-cutting requirements

1. **No horizontal page scroll** at any width down to 320px.
2. **Tap targets ≥ 44px** for nav links, the row/card expand control, and buttons.
3. **Inputs ≥ 16px font-size** to prevent iOS zoom-on-focus.
4. Preserve the existing dark theme, accent color, and all interactive JS behavior
   (sorting, filtering, expand/collapse) — only layout/presentation changes.

## Per-page design

### Shared chrome — `style.css`
- **Nav:** on narrow screens becomes a single horizontal scroll strip (no wrapping /
  overflow of the four links); active-link accent preserved; links sized to ~44px tap
  height. Hide scrollbar where possible.
- **Header:** reduce `h1`/tagline sizes; `.stats` collapse to one column with tighter
  padding.
- Keep the existing 12px container side padding.

### Index — `index.css` (combos table → stacked cards)
- Below 600px, the `<table>` renders as bordered cards via the `display:block`
  technique (`table`, `thead`, `tbody`, `tr`, `td` all `display:block`).
- `thead` hidden in card mode. Cards are pre-sorted by rank (default count-desc);
  sorting/filtering JS still operate on the same DOM.
- Card layout:
  - `td.rank` → small accent badge, top of card.
  - `td.drivers` → prominent line, drivers allowed to wrap.
  - `td.count` → labelled row, `::before { content: "Podiums" }`.
  - `td.last` → labelled row, `::before { content: "Last" }`; race name shown.
  - `td.expand` → clear tap affordance; whole card remains tappable to expand.
- Detail (expanded) row renders full-width beneath its card.
- Filters: search inputs stack vertical, full-width, ≥16px font; Clear button
  full-width below.

### Seasons — `seasons.css`
- Tighten the already-wrapping season header zones.
- Per-race table (`.races-table`) converts to the same stacked-card treatment below
  600px: round + race name as the card heading; `.col-drivers` and `.match-depth` as
  labelled lines. No sideways scroll.

### Soulmates — `soulmates.css`
- Already stacks at 960px. Refine for phones: tighter `.pl-row` padding, allow
  `.pl-names` to wrap (remove ellipsis truncation on mobile), fact cards full-width
  with comfortable spacing.

### Charts — `charts.css`
- Reduce `.chart-card` padding on mobile; ensure chart hosts stay `width:100%`
  (ApexCharts is responsive). Condense/drop the years column in `.pair-row` on narrow
  screens so rows don't overflow.

## Out of scope (YAGNI)
- No JS changes, no build-pipeline changes, no new dependencies.
- No mobile-specific re-sorting UI (cards are pre-sorted; desktop retains header
  sorting).
- No data or content changes.

## Testing / verification
- Visually verify each page at 320px, 375px, and 414px widths (and confirm desktop
  layout is unchanged at ≥ 720px) using browser devtools / responsive mode.
- Confirm: no horizontal scrollbar; expand/collapse, filtering, and sorting still work;
  inputs do not trigger zoom on focus.
